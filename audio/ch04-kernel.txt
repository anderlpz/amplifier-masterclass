The kernel — amplifier-core — is deliberately small, and it writes zero bytes to disk. Inside it are five components. Each exists because every module in the system depends on it. Five things live in the kernel. Nothing else.

First: Session Lifecycle. A session is the container around a particular instance of the system running. Four phases: Create, Initialize, Execute, Cleanup. Fixed loading order. Orchestrator and Context Manager are required. Everything else is optional. Cleanup runs in a finally block. Always. Every session passes through these four phases — Create assembles configuration, Initialize loads modules in fixed order, Execute handles message-response turns, and Cleanup always runs, even after errors. Chapter Eight walks through each phase in detail.

Second: Contracts. Six module types, each defined as a contract. If a module has the right methods, it satisfies the contract. No base class required. No explicit registration. Small, stable contracts that rarely change. When they do change, every module of that type is affected, which is why stability matters here more than anywhere else.

Third: Module Loader. Discovers available modules, loads them, and gives each one a chance to set itself up. The loader doesn't need to know what type of module it's loading. It calls mount, and the module takes care of the rest.

Fourth: the Coordinator. Despite its name, the Coordinator does not coordinate anything. It's a passive registry — think of it as a phone book. Three operations: mount, get, unmount. When a module mounts, it goes into the right slot. When something needs to find a module, it checks the Coordinator. No decisions. No message routing. No behavior. The actual coordinating is done by the Orchestrator, which is a module, not part of the kernel.

Fifth: Hooks. Hooks are injection points for deterministic, non-AI-driven actions. When something happens — a tool is called, a session starts, a file is written — hooks fire automatically. The AI model has no idea they exist. There are forty-one canonical events, organized by category. Events and hooks are one system — a single dispatch mechanism. It runs on every tool call, every message, every session transition.

And finally, the defining constraint: zero file I/O. No log files, no transcripts, no configuration files. Logging happens through hook modules. Persistence happens through context managers. Configuration loading happens in the Foundation layer, entirely outside the kernel.
