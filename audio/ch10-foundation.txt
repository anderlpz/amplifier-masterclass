Foundation — amplifier-foundation — turns text bundle files into running sessions. It is the Amplifier team's reference implementation: their opinions about how to use the kernel, packaged for others. But here's the key: the kernel does not need it. You could build your own equivalent.

Think of Foundation as a convenience layer. The kernel accepts a raw configuration dictionary and a set of module paths. Foundation saves you from writing that dictionary yourself. It reads your bundle files, resolves your module references, installs packages, and assembles the configuration the kernel needs.

Foundation's output is called a PreparedBundle, and it has four parts: a mount plan saying which modules to load in what slots with what configuration, a resolver that maps module names to file paths, the original bundle preserved for reference, and package paths for cleanup.

The preparation pipeline handles all the practical mess between a text file and a running session: creating isolated environments, downloading modules from registries, resolving version constraints, assembling paths. Eight steps from source file to running session — compile a mount plan from the configuration, set up the environment including virtual environment creation, install required Python packages, collect module specifications, download and install missing modules, persist installation state to avoid re-downloading, create the resolver function, and package everything into the PreparedBundle. The kernel never sees any of this work. It receives a clean mount plan and a resolver.

But Foundation doesn't just set up a session and disappear. It registers two callbacks that remain active for the session's entire lifetime. The System Prompt Factory is called on every single turn — it re-reads all at-mention files, resolves nested references up to three levels deep, deduplicates content by hash, and rebuilds the complete system prompt. This is the mechanism behind the live re-reading we discussed in the previous chapter. The BundleModuleResolver is called on demand during module loading, mapping module identifiers to actual file paths.

Neither Foundation nor the kernel has a code dependency on the other. The kernel calls functions without knowing who provided them. Foundation calls the kernel's API without accessing kernel internals. Connected at runtime through callbacks. Independent at compile time through interfaces.
