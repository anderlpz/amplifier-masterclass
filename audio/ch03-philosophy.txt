Three ideas shape how Amplifier is built. They weren't written on a whiteboard first. They emerged from the code, from decisions made repeatedly about what belongs in the kernel and what doesn't.

The first principle is "mechanism, not policy." The kernel fires a tool-pre event before every tool call. A safety module decides whether to block it. A logging module records it. The kernel has no opinion about what should happen. It fires events and collects results. This principle comes from operating system design. The Linux kernel provides process scheduling and memory management without deciding how programs use those mechanisms. Amplifier does the same: the kernel provides module loading, event dispatch, and session lifecycle. It never decides which modules matter or what should happen during a session. When rules change — new safety requirements, new compliance needs — you change modules. The kernel stays the same.

The second principle is "the center stays still." A new AI model ships. Someone writes a provider module. The kernel does not know or care which model it talks to. New tools, new paradigms, new safety requirements: all handled at the edges. The center does not move. The center stays still so the edges can move at any speed.

The third principle is "ruthless simplicity." Search the entire amplifier-core codebase for file-writing operations: zero matches. If something can live in a module, it must. Every kernel line constrains every module's future. Logging, persistence, and configuration are policy — they live in modules, where they can be independently replaced.

These three principles come to life in a practical example. Consider what happens when you need human approval for certain operations. In a system without this separation, you'd modify the tool-calling code itself — add an if-statement, check a config flag. The approval logic gets woven into the execution logic. Every future tool must remember to include the approval check. Miss one, and the policy has a hole. In Amplifier, the kernel fires a tool-pre event. A hook module catches it. If approval is needed, it returns "ask user." The tool itself has no knowledge that approval was required. The policy lives in a module; the mechanism lives in the kernel. This pattern repeats at every level: logging is a hook, context injection is a hook response, safety rules are hook policies. Each can be independently developed, tested, and replaced.
