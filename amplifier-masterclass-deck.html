Chapter 1 · Introduction

Amplifier is a framework for building your AI harness.

A harness takes raw power and gives it direction. An AI model handles reasoning. Everything around the model — tools, memory, rules, communication — needs structure. Amplifier provides that structure.

Think of it as a construction kit: it gives you the pieces, and you assemble the harness that fits your use case.

The OS Analogy

Like an operating system has a tiny kernel at its center — managing memory, scheduling tasks — with everything else plugging in through well-defined interfaces. Amplifier borrows this structure.

Modular by Design

New AI model? Write a provider module. Need safety rules? Add a hook module. The core does not change. It provides infrastructure; the modules provide behavior.

Chapter 3 · Design Philosophy

Three ideas shape how Amplifier is built.

They emerged from the code, from decisions made repeatedly about what belongs in the kernel and what doesn't.

Mechanism, not policy

The kernel fires events. Modules decide what happens. When rules change, you change modules. The kernel stays the same.

The center stays still

New models, new tools, new paradigms — all handled at the edges. The center does not move. So the edges can move at any speed.

Ruthless simplicity

Zero file I/O in the kernel. If it can live in a module, it must. Every kernel line constrains every module's future.

Despite its name, the Coordinator does not coordinate. It is a registry — a shelf with labeled slots.
The actual coordinating is done by the Orchestrator, which is a module, not part of the kernel.

Chapter 4 · The Universal Contract

One contract for every module type.

async def mount( coordinator: Coordinator, config: dict ) -> Optional[cleanup_fn]: # receive the registry + configuration # do your setup work # register yourself in the coordinator # optionally return a cleanup function

Structural subtyping

No base class required. If your class has the right methods with the right signatures, it qualifies. Contracts are small and stable.

Zero file I/O

Search the entire amplifier-core codebase for file-writing operations: zero matches. Logging, persistence, configuration — all live in modules.

Chapter 6 · The Orchestrator

The one module with runtime access to everything.

The privilege gap

Every module gets the Coordinator once, at mount time. The Orchestrator gets it again, at runtime, every time it's called. It can reach into the registry and access any module: Context Manager, Provider, Tool, Hook Handler.

No other module can do this. A Provider can't look up which Tools are available. A Tool can't check which Provider is being used. By design, not accident.

The loop lives outside the kernel

The agent loop — "think, act, observe, repeat" — is not in the kernel. It lives in an Orchestrator module. The kernel provides the execution contract; the loop itself is policy.

A different Orchestrator module could implement parallel provider calls, multi-step planning, or debate between models — without the kernel changing.

Every module connects once at mount. The Orchestrator stays connected at runtime.
Chapter 6 · One Conversation Turn

What happens when you send a message.

1 Message in — the session receives it and hands it to the Orchestrator
2 Hooks fire — context monitoring begins, schema/docs injected
3 Fetch context — Orchestrator calls Context Manager for conversation history
4 Call LLM — Provider sends assembled context to the AI model
5 Tool check — if the AI wants a tool, hooks gate first (tool:pre)
6 Tool runs — hooks inject feedback after (tool:post)
7 Response out — rich data flows through events; if tool called, loop back to step 2

inject_context is the most powerful hook action. A hook places a message into the conversation that the AI reads on its next turn, as if it were part of the natural flow. The AI does what language models do: read text and respond to it.

Chapter 7 · The Most Important Distinction

Tools vs Hooks

Tools

The AI decides
  • - Listed in the system prompt
  • - AI sees name, description, schema
  • - AI chooses when to call
  • - Calls appear in conversation history
  • - Extend what the model can do

Hooks

Code decides
  • - Not in the system prompt
  • - AI cannot see or call them
  • - Fire automatically on events
  • - Can observe, block, inject, or modify
  • - Govern what the model may do

Tools are capabilities you hand to the AI. Hooks are policies you enforce around it.
Use tools when you want the AI to decide. Use hooks when the outcome must be guaranteed.

Chapter 7 · Hook Priority Cascade

When multiple hooks respond, priority resolves.

1
deny
Blocks everything. One deny stops the operation. Fail-closed security.
2
ask_user
Pauses execution for a human decision. First one wins.
3
inject_context
Accumulates. Multiple hooks can all inject messages. All merged into conversation.
4
modify
Transforms data. Multiple modify hooks chain — each sees the previous output.
5
continue
No-op passthrough. "I've seen this event and have nothing to add."

Deny is first for a reason — it doesn't matter if ten hooks say allow. One deny stops it. Circuit breaker.

Child sessions linked by parent_id only. No shared modules, no shared memory, no automatic inheritance. Each session is fully self-contained.

File output comes from modules

When you see transcript.jsonl or events.jsonl — the kernel didn't write them. Hook modules did. Remove every hook, and the session still runs identically.

Chapter 9 · Bundles and Configuration

Text files. No code required.

--- bundle: model: anthropic/claude tools: - name: filesystem - name: web-search hooks: - name: safety-guard --- You are a helpful coding assistant. You write clean, well-documented code. You follow the project's coding standards (see @project:standards.md).

Composition

Bundles stack via include:. Last wins. Session settings deep-merge. Module lists merge by key. Instruction text replaces entirely.

@mentions

Reference external files with @namespace:path/to/file.md. Recursive up to 3 levels. Deduplicated by SHA-256 hash.

Live re-reading

Context files are re-read on every turn. Edit while an agent runs — it sees the change on its next turn. No restart required.

Chapter 10 · The Foundation Bridge

How text files become running sessions.

Foundation (amplifier-foundation) is optional. The kernel works without it. It's a convenience layer that translates bundles into the raw configuration dictionary the kernel needs.

8-Step Pipeline

Compile mount plan → setup environment → install packages → collect module specs → download modules → persist state → create resolver → package bundle

System Prompt Factory

Called on every turn. Re-reads all @mention files, resolves nested references, deduplicates, rebuilds the complete system prompt. This is why @mentions are live.

Runtime callbacks

Foundation doesn't disappear after setup. Two persistent channels remain active: System Prompt Factory (every turn) and BundleModuleResolver (on demand).

Connected at runtime through callbacks. Independent at compile time through interfaces.

Chapter 11 · Higher-Order Primitives

Agents, Skills & Recipes

No new kernel support. Built entirely from sessions, bundles, and modules.

Agents are bundles

Same file format, same parser. Only difference: YAML key is meta: instead of bundle:. The kernel has no concept of "agents" — it runs sessions with configurations.

Skills

Markdown files the agent discovers on demand. Three-level progressive disclosure: metadata (~100 tokens), full content (~1–5k), companion files (0 until read).

Recipes

YAML playbooks: chained sessions with approval gates and checkpoint resumability. Output from step N becomes input to step N+1.

Modes

Runtime behavior overlays. Each tool can be set to safe, warn, confirm, or block. Built from hooks and inject_context.

Model routing

Agents declare the type of thinking needed: coding, reasoning, fast, vision. A routing matrix maps roles to concrete models.
